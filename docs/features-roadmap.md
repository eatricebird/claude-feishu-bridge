> 目标：实现 80% 编程场景的远程覆盖，让程序员在离开电脑后仍能指挥 AI 工作

### 1.3 现有架构局限

- **单向通信**：只能从 Claude 推送通知到飞书
- **被动响应**：用户只能对特定事件做出是/否选择
- **事件单一**：仅支持 PermissionRequest 事件

## 二、需要实现的核心功能

### 2.1 AskUserQuestion Hook 支持

**场景描述**：当 AI 需要向用户提问时（如选择技术栈、确认配置、澄清需求），转发到飞书并支持用户选择响应。

**实现要点**：
- 新增 `src/hooks/ask_user_question.py` hook
- 飞书卡片支持多种交互形式：
  - 单选按钮（选择技术栈、框架等）
  - 多选按钮（选择需要的功能模块）
  - 文本输入（补充说明、配置值）
- 将用户选择通过 stdout 返回给 Claude Code

**飞书卡片示例**：
```
┌─────────────────────────────────────┐
│  需要您的选择                        │
├─────────────────────────────────────┤
│  问题: 使用哪个状态管理库？          │
│                                      │
│  ○ Redux (推荐用于大型应用)          │
│  ○ Zustand (轻量级选择)              │
│  ○ Context API (简单场景)            │
│                                      │
│  [确认选择]  [取消]                  │
└─────────────────────────────────────┘
```

### 2.2 ExitPlanMode Hook 支持

**场景描述**：当 AI 完成计划制定等待批准时，在飞书显示完整计划并支持批准/修改/拒绝。

**实现要点**：
- 新增 `src/hooks/exit_plan_mode.py` hook
- 飞书卡片展示计划内容（可能需要分页或折叠展开）
- 支持三种操作：
  - 批准 - 直接开始执行
  - 拒绝 - 取消当前计划
  - 修改意见 - 输入修改建议后重新生成

**飞书卡片示例**：

**场景描述**：当任务完成、失败或需要人工介入时，推送通知到飞书，保持用户对工作进度的感知。

### 2.4 双向通信机制（关键功能）

**场景描述**：用户可以在飞书中主动发送消息，注入新的提示词到正在运行的 Claude 会话，实现真正的远程控制。

**实现要点**：
- 飞书接收用户文本消息（通过 `im.message.receive_v1` 事件）
- 将消息注入到 Claude Code 输入流
- 实现方案（需探索）：
  - **方案 A**：通过 FIFO 命名管道通信
  - **方案 B**：模拟终端输入（expect 脚本）
  - **方案 C**：Claude Code 提供外部输入 API（需官方支持）

**使用场景**：
- 修改之前的指令
- 提供补充信息
- 中止当前任务
- 切换工作方向

### 2.5 错误恢复决策

**场景描述**：当 AI 遇到错误需要用户指示如何处理时（如测试失败、编译错误、依赖冲突），在飞书展示错误并请求决策。

**实现要点**：
- 新增 `src/hooks/error_recovery.py` hook
- 解析错误信息，提取关键内容
- 提供常见处理选项：
  - 重试（可能添加 --force 或绕过缓存）
  - 跳过（继续执行后续步骤）
  - 修改后继续（让 AI 修复错误后重试）
  - 中止（完全停止执行）

**飞书卡片示例**：

## 五、技术挑战与解决方案

### 5.1 提示词注入机制

**挑战**：Claude Code 本身不支持外部输入注入

**可能方案**：

| 方案 | 可行性 | 优缺点 |
|------|--------|--------|
| FIFO 命名管道 | 中 | 需要修改 Claude Code 输入逻辑 |
| expect 脚本模拟 | 高 | 实现复杂，易受终端变化影响 |
| tmux/屏幕注入 | 高 | 需要特定运行环境 |
| 官方 API 支持 | 待定 | 最理想方案，需要向官方提需求 |

**推荐方案**：先使用 FIFO 管道 + 轮询检测

```python
# 示例：FIFO 管道实现
# 创建管道
os.mkfifo("/tmp/claude_input_fifo")

# Hook 脚本定期读取
with open("/tmp/claude_input_fifo", "r") as f:
    f.setblocking(False)
    while True:
        try:
            user_input = f.readline()
            if user_input:
                # 将输入注入到 Claude
                inject_to_claude(user_input)
        except BlockingIOError:
            time.sleep(0.5)
```

### 5.2 消息上下文保持

**挑战**：长时间离开后，用户需要知道当前进度和上下文

**解决方案**：
- 在飞书卡片中保留历史操作记录
- 提供"查看当前状态"按钮，获取最新上下文
- 重要节点自动推送状态摘要

### 5.3 并发多个会话

**挑战**：同时运行多个 Claude 会话时的消息路由

**解决方案**：
- 为每个会话分配唯一 session_id
- 在存储层维护 session -> 飞书消息映射
- 支持在飞书切换不同会话

### 5.4 飞书卡片限制

**挑战**：卡片内容长度、交互复杂度的限制

**解决方案**：
- 长内容使用折叠/展开
- 分多张卡片展示，通过"下一页"按钮导航
- 详细内容使用飞书文件，卡片中只显示摘要

## 七、评估标准

完成以上功能后，预期可以达到：

- **覆盖场景**：约 70-80% 的编程交互场景
- **响应延迟**：通知推送 < 3秒，用户响应 < 5秒生效
- **可用性**：无需在电脑前即可完成大多数决策
- **安全性**：敏感操作仍需二次确认

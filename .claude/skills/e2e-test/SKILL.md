# 飞书桥接端到端测试

## 开始测试

当用户调用 `/e2e-test` 时：

1. 显示欢迎信息和测试场景列表
2. 使用 AskUserQuestion 让用户选择场景
3. 根据选择执行对应场景
4. 每个场景完成后，询问是否继续


## 重要原则：延迟清理

**测试过程中不要执行 `rm` 删除文件**，因为会触发额外的 PermissionRequest Hook，打断测试流程。

- 每个场景产生的测试文件（`/tmp/e2e_*`）只做记录，不立即删除
- 所有清理统一在测试结束后或用户中途退出时执行
- 清理时使用单条 `rm -f /tmp/e2e_*.txt` 一次完成

## 测试场景

### 场景 1：权限审批 - 批准

**目标：** 验证用户在飞书中批准权限请求后，Claude 能正确执行操作。

**执行步骤：**

1. 显示场景标题："─────────────────────────────────────────────────────────────\n场景 1：权限审批 - 批准\n─────────────────────────────────────────────────────────────"

2. 告诉用户："即将测试权限批准流程。我会请求创建一个测试文件。请检查飞书，点击「允许」按钮。"

3. 执行测试命令：`echo "E2E Test $(date)" > /tmp/e2e_approve_test.txt`
   - 这将触发 PermissionRequest Hook
   - Hook 会向飞书发送权限卡片

4. 等待用户在飞书中响应后，使用 AskUserQuestion 询问："已在飞书中点击「允许」了吗？"
   - 选项：["是，已完成", "取消"]

5. 如果用户取消，显示"场景已取消"并返回场景选择

6. 验证结果：
   - 使用 Read 工具读取 `data/permissions.json`
   - 找到最新的请求（按 created_at 排序）
   - 检查 `status` 是否为 "allow"
   - 使用 Bash 检查文件是否存在：`ls -la /tmp/e2e_approve_test.txt`

7. 显示结果：
   ```
   ╔════════════════════════════════════════════════════════════╗
   ║  ✓ 场景 1 通过：权限审批-批准流程正常                       ║
   ║                                                            ║
   ║  验证结果：                                                ║
   ║  • permissions.json 状态: allow                           ║
   ║  • 测试文件已创建                                         ║
   ╚════════════════════════════════════════════════════════════╝
   ```

8. 询问继续：使用 AskUserQuestion "继续下一个场景？"
   - 选项：["继续", "退出测试"]
   - 如果继续，返回场景选择
   - 如果退出，显示测试摘要并结束

### 场景 2：权限审批 - 拒绝

**目标：** 验证用户在飞书中拒绝权限请求后，Claude 正确拒绝操作。

**执行步骤：**

1. 显示场景标题

2. 首先创建一个测试文件（用于后续删除测试）：
   - `echo "Test file" > /tmp/e2e_deny_test.txt`

3. 告诉用户："即将测试权限拒绝流程。请检查飞书，点击「拒绝」按钮。"

4. 执行测试命令：`rm -f /tmp/e2e_deny_test.txt`
   - 触发 PermissionRequest Hook

5. 等待用户响应，询问："已在飞书中点击「拒绝」了吗？"

6. 验证结果：
   - 读取 `data/permissions.json`
   - 检查最新请求的 `status` 是否为 "deny"
   - 检查文件是否仍然存在（操作被拒绝）：`ls -la /tmp/e2e_deny_test.txt`

7. 显示结果：
   ```
   ╔════════════════════════════════════════════════════════════╗
   ║  ✓ 场景 2 通过：权限审批-拒绝流程正常                       ║
   ║                                                            ║
   ║  验证结果：                                                ║
   ║  • permissions.json 状态: deny                            ║
   ║  • 操作被正确拒绝                                         ║
   ╚════════════════════════════════════════════════════════════╝
   ```

8. 询问继续

## 验证方法

每个场景使用以下方式验证：

1. **读取存储文件**：使用 Read 工具读取 `data/permissions.json` 和 `data/last_answer.json`

2. **解析 JSON 数据**：
   ```python
   # 找到最新的请求
   latest = max(data.items(), key=lambda x: x[1].get('created_at', 0))
   status = latest[1].get('status')
   ```

3. **检查文件操作结果**：使用 Bash ls/cat 验证文件状态

4. **显示验证结果**：向用户报告每个验证点

## 用户交互模式

- **场景选择**：使用 AskUserQuestion 展示场景列表
- **进度确认**：每个场景后使用 AskUserQuestion 确认是否继续
- **随时退出**：支持用户选择"退出测试"
- **状态显示**：实时显示当前步骤和预期操作

## 错误处理

如果验证失败：

1. 显示错误详情：具体哪个验证点失败

2. 提供诊断信息：
   - 读取 `data/permissions.json` 显示实际状态
   - 显示 webhook 日志：`data/webhook_test.log`

3. 询问用户：使用 AskUserQuestion "是否继续其他场景？"
   - 选项：["继续", "退出测试"]

## 注意事项

1. **执行前检查**：
   - 确认 webhook 服务器正在运行
   - 确认飞书应用配置正确
   - 确认网络连接正常

2. **场景顺序**：
   - 建议按顺序执行（1 → 2 → 3 → 4 → 5）
   - 用户可自由选择场景

3. **数据隔离**：
   - 所有测试文件使用 `/tmp/e2e_*` 前缀
   - 测试结束后统一清理（不中途删除，避免触发权限请求）

4. **配置修改**：
   - 只有场景 5 会修改配置
   - 场景 5 结束后立即恢复原始值（不需要等到最后）
